***** [ ] It is asserted that "Fast Refresh" names the focal structure
   It is described that Fast Refresh is Next.js’s implementation of React’s Hot Module Replacement (HMR) designed to preserve component state while applying code changes. It is said that this mechanism updates only the modified modules in development without a full reload, providing faster iteration and feedback.
   *** Diagrams
   #+BEGIN_SRC plantuml
   @startuml
   [Code Change] --> (Fast Refresh)
   (Fast Refresh) --> (Update Module in Place)
   (Fast Refresh) --> (Preserve Component State)
   (Fast Refresh) --> (Fallback to Full Reload) : if error
   @enduml
   #+END_SRC

***** [ ] It is said that Fast Refresh rules ensure correctness
   It is explained that Fast Refresh preserves React state when edits are safe, but falls back to a remount when unsafe changes occur. It is noted that safe edits include function body changes, while unsafe edits include altering component signatures or exports. This balance ensures developer convenience without compromising correctness.
   *** Diagrams
   #+BEGIN_SRC plantuml
   @startuml
   [Edit Component] --> (Check Safety)
   (Check Safety) --> (Preserve State) : if safe
   (Check Safety) --> (Remount Component) : if unsafe
   @enduml
   #+END_SRC

***** [ ] It is observed that Fast Refresh integrates with error recovery
   It is mentioned that when runtime errors occur, the system overlays an error screen but preserves React state once the issue is fixed. It is suggested that this reduces disruption in developer workflow and allows quicker debugging.
   *** Diagrams
   #+BEGIN_SRC plantuml
   @startuml
   [Runtime Error] --> (Error Overlay)
   (Error Overlay) --> (Preserve State After Fix)
   (Error Overlay) --> (Recover with Fast Refresh)
   @enduml
   #+END_SRC

***** [ ] It is asserted that "Next.js Compiler" names the focal structure
   It is described that the Next.js Compiler is powered by SWC (Speedy Web Compiler), a Rust-based compiler replacing Babel and Terser. It is said that the compiler provides faster JavaScript/TypeScript transpilation, minification, and support for modern syntax features like JSX and TypeScript.
   *** Diagrams
   #+BEGIN_SRC plantuml
   @startuml
   [Source Code] --> (SWC Compiler)
   (SWC Compiler) --> (Transpile JS/TS)
   (SWC Compiler) --> (Transform JSX)
   (SWC Compiler) --> (Minify Output)
   @enduml
   #+END_SRC

***** [ ] It is said that Compilation and Transformation define the core pipeline
   It is noted that SWC parses source files into an AST (Abstract Syntax Tree), then applies transformations for JSX, TypeScript, and other language features. It is suggested that the compiler also performs tree-shaking and dead-code elimination for production builds.
   *** Diagrams
   #+BEGIN_SRC plantuml
   @startuml
   [SWC Compiler] --> (Parse Source to AST)
   (Parse Source to AST) --> (Apply Syntax Transforms)
   (Apply Syntax Transforms) --> (Optimize & Minify)
   (Optimize & Minify) --> (Emit Final Bundle)
   @enduml
   #+END_SRC

***** [ ] It is observed that Performance and Extensibility are emphasized
   It is explained that the Next.js Compiler runs significantly faster than Babel due to its Rust implementation. It is indicated that SWC is extensible and forms the foundation for experimental features such as the React Compiler integration, and plays a role in the broader Next.js ecosystem including bundling, transforms, and performance tooling.
   *** Diagrams
   #+BEGIN_SRC plantuml
   @startuml
